# 2026-07-19 — M30.1: stars "reset" — diagnosis + durability (loop/20260719-m30-stars)

## Diagnosis (investigation agent, full-day diff trace)

**No code regression.** Stars and 🎧 badges live in ONE storage object
with one key template (`${setId}.${tense}.${mode}`) read through one
accessor — no commit shipped today (or ever) can zero one and spare the
other. MODES/TENSES/CONTRAST_KEY are frozen (git -L proof); the beacon
never writes localStorage; the SW touches only the Cache API.

The symptom is a storage-context event. Ranked triggers:
1. **iOS installed-app container** — an installed Home-Screen app on
   iOS keeps storage SEPARATE from Safari by system design; a fresh
   install starts at zero.
2. **WebKit 7-day ITP eviction** — script-writable storage is erased
   for rarely visited, non-persisted origins; we never called
   `persist()` on boot (only inside Descargas).
3. **`?m18demo=1` rounds** — demo mode deliberately records nothing.

"Badges intact" is most plausibly the STATIC 🎧 iconography (set/study
screens and the Escucha card show 🎧 unconditionally; EARNED badges on
home cards require listenBadges>0 — genuinely selective loss is
impossible under this storage model).
