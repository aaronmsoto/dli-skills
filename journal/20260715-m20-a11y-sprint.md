# M20 a11y sprint: owner-reported contrast + phantom-selection fixes · 2026-07-15 · claude/github-repo-context-status-q2u4i8

Owner live-review findings after the M18 release, fixed same-day (session
branch, owner-directed).

- Goal criteria addressed: all of M20 (M20-1/2/3 + V).
- What was done:
  - **M20-1** done-card ink: styles.css fades `.match-card.done` to opacity
    .75, washing text below 4.5:1 in both themes. Prado layer now sets
    opacity 1 + `color: var(--good-ink)` (5.56:1 light / 6.02:1 dark on
    --good-bg), keeping the settled look via ink instead of transparency.
  - **M20-2** primary-button ink: `.h-en` support copy on green primary
    buttons was --ink-soft gray — measured 1.04:1 (light) and 1.11:1 (dark),
    effectively invisible. Investigating exposed a PRE-EXISTING dark-theme
    failure: white on --brand #74c489 = 2.10:1 on every primary button (M17's
    axe gate snapshots empty screens; results buttons were never measured in
    dark). New token `--btn-primary-ink` (light #fff 6.29:1, dark #243026
    6.55:1) in all three theme blocks; `.btn.primary` + the results-screen
    variant consume it; `.h-en{color:inherit}` on primary buttons.
  - **M20-3** vuelo phantom selection: freshly dealt clouds under a
    touch-parked pointer painted :hover as a lingering selection. Applied the
    app's established sticky-hover pattern: `sky.classList.add("no-hover")`
    on each deal, cleared on the first real pointermove, with the matching
    `.vuelo-sky.no-hover .vuelo-cloud:hover` transparent-border rule.
- Validation: npm test 52/52 · e2e PASS with the new M20 block running the
  contrast assertions in BOTH themes (dark via seeded settings.theme) and
  replaying the sticky-hover scenario with a real parked mouse pointer.
- Decisions & rationale: ink-over-transparency for "done" styling (opacity
  washes any token); token-based button ink so future theme changes stay
  auditable; sticky-hover fix reuses the exact M13-era pattern for
  consistency.
