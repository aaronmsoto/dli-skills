# 2026-07-19 — M25.2: service worker core (loop/20260719-m25-sw-core)

## What

Second M25 (PWA) iteration — the offline shell, exactly per SPEC §5.5:

- **`sw.js`** (root, hand-written, joins the payload budget):
  - `VERSION = "shell-v1"` names the shell cache; bump on shell changes.
  - install: `cache.addAll` of the 17-file shell (2 HTML + manifest +
    3 CSS + 10 JS modules + audio/manifest.json) + `skipWaiting()`.
  - activate: deletes every cache that is neither `VERSION` nor an
    `audio-g*` group cache, then `clients.claim()`.
  - fetch: same-origin GETs only. `.mp3` → CACHE-FIRST lookup across the
    `audio-gNN` caches (this worker NEVER populates them — that is
    Descargas' job, M25.3), falling through to network. Navigations →
    network-first; on network failure ONLY, cached shell with
    `ignoreSearch: true` (so `/?m18demo=1` reaches the app offline) —
    online, querystrings pass through untouched. Everything else →
    network-first with cache fallback (freshness beats speed on a
    no-hash site; SWR rejected in the design for version-skew risk).
- **js/app.js** — registration after boot, gated on
  `"serviceWorker" in navigator && !navigator.webdriver &&
  !sessionStorage.getItem("conjuga.noSW")`, so the e2e harness and its
  manifest-204 stubs stay SW-free and deterministic.

## Tests

- Existing e2e blocks untouched (webdriver gate keeps them SW-free).
- New dedicated block in its own browser context (SW registrations are

  `location.search` → back online the query still passes through → a
  second page with `conjuga.noSW` set asserts the gate blocks
  registration → unregister + delete all caches on the way out.

  sw.js now inside the 100 KB gz payload budget.
